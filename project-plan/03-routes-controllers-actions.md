# 03 — Routes, Controllers & Actions

Scaffold every route the app needs across the three surfaces defined in [architecture.md](architecture.md), wire them to thin controllers, and stub out every Action they call. **Action bodies are left empty (`// TODO`) in this step** — this section is about creating the full structure. Later sections fill the Actions in.

**Architecture note:** the whole app is a Vue 3 SPA authenticated via Laravel Sanctum (cookie-based). `web.php` holds only SPA shell routes and signed downloads. All app functionality lives under `/api/v1/*`.

---

## 0. Route files + middleware

```
routes/
├── web.php            # SPA shell, Sanctum CSRF cookie, signed file downloads
├── api/
│   └── v1.php         # all app functionality as JSON (Sanctum SPA auth)
└── rest/
    └── v1.php         # webhooks, prefix /rest/v1, no CSRF
```

`bootstrap/app.php`:

```php
->withRouting(
    web: __DIR__.'/../routes/web.php',
    api: __DIR__.'/../routes/api/v1.php',
    apiPrefix: 'api/v1',
    commands: __DIR__.'/../routes/console.php',
    then: function () {
        Route::middleware('rest')
            ->prefix('rest/v1')
            ->group(base_path('routes/rest/v1.php'));
    },
)
->withMiddleware(function (Middleware $middleware) {
    $middleware->statefulApi();                  // Sanctum SPA mode
    $middleware->alias([
        'role'              => \App\Http\Middleware\EnsureRole::class,
        'right'             => \App\Http\Middleware\EnsureRight::class,
        'manager.scope'     => \App\Http\Middleware\InjectManagerScope::class,
        'webhook.signature' => \App\Http\Middleware\VerifyWebhookSignature::class,
    ]);

    $middleware->group('rest', [
        \App\Http\Middleware\VerifyWebhookSignature::class,
    ]);
})
```

`config/sanctum.php` stateful domains include the app host (configured via `SANCTUM_STATEFUL_DOMAINS=app.test,localhost`). `config/cors.php` allows credentials.

Middleware aliases:
- `role:superadmin|manager|customer` — requires one of the given roles (uses `User::hasRole`)
- `right:tickets.update` — requires the right (uses `User::hasRight`)
- `manager.scope` — loads `manager_group_scopes` → request `assigned_group_ids`
- `webhook.signature` — verifies Freshdesk webhook HMAC before the controller runs

## 1. Web routes — `routes/web.php`

Shell-only. Laravel serves no app HTML beyond two Blade entry files that boot Vue.

| Method | Path | Controller@method | Purpose |
|---|---|---|---|
| GET  | `/`                       | `Web\SpaShellController@root`     | Redirect to `/dashboard` or `/portal` based on session role; guests go to `/portal/login` |
| GET  | `/dashboard/{any?}`       | `Web\SpaShellController@dashboard`| Returns `resources/views/apps/dashboard.blade.php` — Vue takes over routing |
| GET  | `/portal/{any?}`          | `Web\SpaShellController@portal`   | Returns `resources/views/apps/portal.blade.php` — Vue takes over routing |
| GET  | `/downloads/{signed}`     | `Web\DownloadController@show`     | Signed-URL endpoint that streams files generated by export actions (`Content-Disposition: attachment`) |

Sanctum auto-registers `GET /sanctum/csrf-cookie`; no explicit route needed.

**Every other request from a browser (login submit, list tickets, create contact, upload import, etc.) goes through `/api/v1/*`.** Deep links in emails (verify, reset, magic link) point to SPA routes — the Vue router reads the token and POSTs to the API.

The `{any?}` wildcards match any subpath (including empty). The controller returns the same Blade shell for every match; the Vue router handles client-side navigation.

## 2. API v1 routes — `routes/api/v1.php`

Prefix `/api/v1`, `api` middleware group (Sanctum stateful + throttle + JSON error handler). Two namespaces: `admin` (superadmin + manager) and `portal` (customer). Plus a shared `auth` namespace.

Two guard flavours share the same endpoints:
- **SPA cookie auth** — `EnsureFrontendRequestsAreStateful` + session, used by the Vue apps.
- **Token auth** — `Authorization: Bearer` against `api_keys.hash`, abilities mapped from `api_keys.scopes`. Used by external integrations.

namespace App\Http\Controllers\Web;

use App\Http\Controllers\Controller;
use Illuminate\Http\RedirectResponse;
use Illuminate\Http\Request;
use Illuminate\View\View;

class SpaShellController extends Controller
{
    public function root(Request $request): RedirectResponse
    {
        $user = $request->user();
        if (!$user) {
            return redirect('/portal/login');
        }
        return $user->hasRole('customer')
            ? redirect('/portal')
            : redirect('/dashboard');
    }

    public function dashboard(): View
    {
        return view('apps.dashboard');
    }

    public function portal(): View
    {
        return view('apps.portal');
    }
}

## 8. Route naming convention

Web shell routes get names so controllers can redirect to them:
- `spa.root`, `spa.dashboard`, `spa.portal`, `downloads.show`

API routes are typically referenced by path from the SPA and don't need `name()`, but we still name auth + webhook routes so they can be built via `URL::signedRoute()` for email deep links:
- `api.admin.auth.login`, `api.admin.auth.reset`
- `api.portal.auth.magic-link.send`, `api.portal.auth.magic-link.consume`
- `api.portal.auth.verify`, `api.portal.auth.reset`
- `rest.webhooks.freshdesk`
